# 存储与上传流水线

## 1. 设计目标

- 任意 SRS/ZLM 节点产生的录像/抓拍，**任意 Upload Worker** 均可读取并上传
- MinIO 上传成功后 **立即删除** GlusterFS 副本，控制缓冲卷容量
- 与现网 `RecordSpace` / `Playback` / `SnapSpace` 模型 **完全兼容**
- 支撑 **133+ 段/秒** DVR 事件（2 万路规模）

## 2. GlusterFS 完整实施

### 2.1 集群拓扑

```
         ┌─────────┐  ┌─────────┐  ┌─────────┐
         │ brick01 │  │ brick02 │  │ brick03 │   gv-playbacks (replica 3)
         └────┬────┘  └────┬────┘  └────┬────┘
              └─────────────┼─────────────┘
                            │
              客户端挂载（SRS / Worker / VIDEO）
```

- **6~9 台** 存储节点，每卷 3 副本
- 播放缓冲卷 `gv-playbacks`：**优先 NVMe SSD**（高 IOPS 小文件）
- 抓拍卷 `gv-snaps`：可 SSD 或 SSD+HDD 分层

### 2.2 卷创建（脚本见 `.scripts/media-cluster/glusterfs/volume-create.sh`）

```bash
# 在所有 brick 节点
gluster peer probe brick02
gluster volume create gv-playbacks replica 3 \
  brick01:/data/brick/playbacks \
  brick02:/data/brick/playbacks \
  brick03:/data/brick/playbacks force
gluster volume start gv-playbacks
gluster volume set gv-playbacks performance.cache-size 256MB
gluster volume set gv-playbacks network.ping-timeout 10
```

### 2.3 目录规范

```
/mnt/easyaiot-media/
├── playbacks/
│   ├── live/{device_id}/{YYYY}/{MM}/{DD}/{timestamp}.flv    # SRS
│   ├── ai/{device_id}/...                                    # SRS AI（若开启 DVR）
│   └── gb28181/{device_id}/...                               # ZLM
├── snaps/
│   └── {device_id}/{timestamp}.jpg                           # 算法临时抓拍
└── staging/                                                  # 可选 batch
```

与现网 SRS 路径兼容：`/data/playbacks/...` → 统一映射为 `/mnt/easyaiot-media/playbacks/...`

### 2.4 客户端挂载

```bash
mount -t glusterfs storage-gv:/gv-playbacks /mnt/easyaiot-media/playbacks
mount -t glusterfs storage-gv:/gv-snaps /mnt/easyaiot-media/snaps
```

`/etc/fstab` 加 `_netdev`；开机 `mount-all.sh` 校验。

### 2.5 容量规划

| 场景 | 计算 | 缓冲容量建议 |
|------|------|--------------|
| 4,000 路 DVR @ 30s | 24MB×(4000/30)≈3.2GB/s 写入 | 保留 24~48h 冗余：**50~100TB** 裸容量（3 副本前） |
| 上传 P99 120s | 3.2GB/s × 120s ≈ 384GB 在途 | 实际上传后秒级删除，在途远小于此 |
| 抓拍 400/s × 200KB | 80 MB/s | snaps 卷 **5~10TB** |

**关键**：Upload Worker 吞吐必须 **≥ DVR 产生速率**，否则 GlusterFS 持续增长。

## 3. Kafka 设计

### 3.1 Topic

| Topic | 分区 | 副本 | 说明 |
|-------|------|------|------|
| `media.dvr.completed` | 32 | 3 | DVR 段完成 |
| `media.snap.completed` | 16 | 3 | 抓拍待上传 |
| `media.dvr.dlq` | 4 | 3 | 死信 |

Partition key：`device_id`

### 3.2 创建脚本

见 `.scripts/media-cluster/kafka/topics.sh`

### 3.3 消费组

- `upload-worker-dvr`：DVR 上传
- `upload-worker-snap`：抓拍上传

Consumer 实例数 ≤ 分区数；建议 **10 Worker × 3 消费线程 = 30 并行**。

## 4. Upload Worker 详细设计

### 4.1 职责

1. 消费 `media.dvr.completed`
2. **文件 stable 检测**（size 连续 3 次采样不变，间隔 1s）
3. 解析 `device_id`（复用 `on_dvr_callback` 匹配逻辑）
4. 确保 `RecordSpace` 存在（调 VIDEO 内部 API 或直连 DB）
5. `fput_object` → MinIO bucket `record-space`，object `{device_id}/{filename}`
6. 可选 **ffmpeg 抽封面** → 上传 thumbnail
7. 写 `Playback` 记录
8. 更新告警 `record_path`（若有关联）
9. **删除 GlusterFS 源文件**
10. 提交 Kafka offset

### 4.2 幂等

- Object key：`{device_id}/{segment_timestamp}.flv`
- 上传前 `stat_object`，若已存在且 etag 一致则跳过上传、仅删本地
- DB `Playback` 用 `(device_id, object_name)` 唯一约束

### 4.3 失败重试

| 错误 | 策略 |
|------|------|
| 文件不存在 | 延迟 5s 重试，最多 12 次 |
| MinIO 5xx | 指数退避，最多 8 次 |
| DB 失败 | 重试 3 次后进 DLQ |
| 超过最大重试 | 写 `media.dvr.dlq` + 告警 |

### 4.4 目录结构

```
MEDIA/services/upload_worker/
├── __main__.py
├── consumer.py
├── handlers/
│   ├── dvr_handler.py      # 从 on_dvr_callback 迁移
│   └── snap_handler.py
├── stable_file.py
└── janitor.py              # 孤儿文件扫描
```

### 4.5 从现网迁移的核心逻辑

来源：`VIDEO/app/blueprints/camera.py` → `on_dvr_callback`

保留：

- `resolve_absolute_playback_path()` 路径解析
- MinIO `fput_object` + `ensure_bucket_public_read_write_policy`
- `Playback` 写入、`file_path_url` 格式
- ffmpeg 封面抽取

移除：

- Flask request 同步处理
- 在 Hook 内等待文件写入（迁至 stable 检测）

### 4.6 Janitor（孤儿清理）

独立线程/ Cron，每 60s：

1. 扫描 `playbacks/` 下 `.flv`，`mtime > 10min`
2. 查 Kafka offset / DB 是否已处理
3. 未处理 → 重新入队或告警
4. GlusterFS 使用率 > 90% → 按 mtime 删最旧（极端保护）

继承：`VIDEO/app/services/playback_disk_guard_service.py`

## 5. MinIO 集群

### 5.1 部署

- 4~8 节点 Erasure Code（EC:4+4 或 8+8）
- 独立 VLAN30，万兆
- 与 GlusterFS **物理分离**

### 5.2 Bucket 策略（保持现网）

| Bucket | 用途 |
|--------|------|
| `record-space` | 监控录像，`{device_id}/{file}` |
| `snap-space` 等 | 抓拍空间 |

### 5.3 生命周期 ILM

- 标准 → 归档（若 `save_mode=1`）
- 按 `RecordSpace.save_time` 天数过期（Worker 或 MinIO ILM 二选一，建议 **MinIO ILM**）

### 5.4 大文件上传

- `>= 64MB` 使用 `minio_client.fput_object` 自动分片
- 并发连接池：`MINIO_POOL_CONNECTIONS=32`

## 6. 抓拍流水线

算法 Worker 写抓拍至 `gv-snaps/{device_id}/{ts}.jpg`，并发送 `media.snap.completed`（或 Worker 扫描目录）。

Handler：

1. 上传 `snap-space` bucket
2. 更新 `SnapSpace` 相关元数据（若有 DB 记录）
3. 删除 GlusterFS 文件

现网参考：`VIDEO/app/services/auto_frame_extraction_service.py`、`snapshot_algorithm_service/run_deploy.py`

## 8. 监控指标

| 指标 | 类型 | 说明 |
|------|------|------|
| `upload_worker_dvr_processed_total` | counter | 成功上传数 |
| `upload_worker_dvr_failed_total` | counter | 失败数 |
| `upload_worker_dvr_latency_seconds` | histogram | 从 Kafka 到 MinIO 完成 |
| `gluster_playbacks_bytes` | gauge | 卷使用量 |
| `gluster_playbacks_file_count` | gauge | .flv 文件数 |
| `kafka_consumer_lag` | gauge | 按 topic |

## 9. hybrid 迁移模式

`MEDIA_UPLOAD_MODE=hybrid` 期间：

- SRS Hook → MEDIA → Kafka（主路径）
- 同时 SRS `on_dvr` 仍指向 VIDEO 旧接口（对比校验）
- 双写一致后关闭旧 Hook

## 10. 数据流小结

```
[SRS DVR 写文件]
       ↓
[GlusterFS 可见于所有 Worker]
       ↓
[Hook → Kafka message]
       ↓
[Worker: stable → MinIO → DB → unlink]
       ↓
[GlusterFS 文件消失，MinIO 为唯一副本]
```
