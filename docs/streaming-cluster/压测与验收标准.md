# 压测与验收标准

## 1. 压测目标（2 万路注册）

| 指标 | 目标值 |
|------|--------|
| 并发推流 | 6,000 路（30%） |
| 并发播放 | 2,000 路 |
| DVR 写入 | 4,000 路常录，≥ 130 段/秒 |
| Upload P99 延迟 | ≤ 120s（段完成 → MinIO 可见） |
| GlusterFS 稳态文件数 | < 50,000 个 .flv |
| Hook P99 响应 | ≤ 50ms |
| 播放首帧 | ≤ 3s（LAN） |
| 故障切换 | SRS 单节点宕机 5min 内 migrate 完成 |

## 2. 压测环境

最低 POC：

- 3× SRS Live + 2× SRS AI + 2× ZLM
- 3× GlusterFS brick
- 1× MinIO（4 节点可缩为 1 节点 POC）
- 3× Kafka，2× Upload Worker
- 1× MEDIA Scheduler

## 3. 压测工具

| 场景 | 工具 |
|------|------|
| RTMP 推流 | `ffmpeg -re -stream_loop -1 -i test.mp4 -c copy -f flv rtmp://...` |
| 批量推流 | 自研 `scripts/load/push_rtmp_batch.py` 或 srs-bench |
| HTTP-FLV 播放 | `ffplay` / wrk + 自定义 |
| DVR 速率 | 统计 Kafka `media.dvr.completed` QPS |
| Upload 延迟 | Worker histogram |

## 4. 压测场景

### 4.1 场景 A：推流容量

- 单 SRS 节点递增推流至 500 路，观察 CPU/带宽/丢帧
- 输出：**单节点安全上限 N**

### 4.2 场景 B：DVR + 上传闭环

- 4,000 路推流 + DVR ON
- 运行 4h
- 验收：GlusterFS 文件数稳定；MinIO 对象数线性增；无 DLQ 堆积

### 4.3 场景 C：播放风暴

- 2,000 路同时 HTTP-FLV 播放
- Edge + SRS 出口带宽监控

### 4.4 场景 D：节点故障

- 随机 kill 1 台 srs-live-05
- 执行 migrate 200 路设备
- 验收：播放恢复 < 5min

### 4.5 场景 E：国标 RTP

- 500 路 GB28181 模拟器 → ZLM
- RTP 端口占用、点播成功率

### 4.6 场景 F：Kafka 堆积恢复

- 停 Worker 30min 制造 lag
- 恢复 Worker，验收 30min 内 lag < 1000

## 8. 已知限制（验收时确认）

- WebRTC 需正确 `candidate` IP，跨 NAT 需 TURN
- GlusterFS 不适合长期存海量小文件，**必须** Upload 及时删除
- RTMP 四层 LB 有限制，推流推荐直连节点 IP
